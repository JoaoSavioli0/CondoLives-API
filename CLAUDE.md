## Build and Run

```bash
./mvnw clean package          # Build JAR
./mvnw spring-boot:run        # Run locally
./mvnw clean test             # Run all tests
./mvnw test -Dtest=ClassName  # Run a single test class
```

Requires Java 21 and a running PostgreSQL instance.

## Database

PostgreSQL with two required extensions: `pgcrypto` (UUID generation) and `citext` (case-insensitive email). The full schema is at `schema_condominio.sql`. Flyway manages migrations; place files as `src/main/resources/db/migration/V*.sql`.

Minimum `application.properties` entries needed to run:
```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/condolives
spring.datasource.username=...
spring.datasource.password=...
spring.jpa.hibernate.ddl-auto=validate
spring.flyway.enabled=true
```

## Architecture

Standard layered Spring Boot app: `controller → service → repository → entity`.

Root package: `com.condolives.api`

Entity sub-packages reflect domain groupings:
- `entity/User/` — `Resident`, `Staff`, `Notification`, `Vehicle`
- `entity/Post/` — polymorphic feed: `Post` (base), `Ticket`, `Suggestion`, `Poll`, `Trade`, `Notice`, plus `Comment` and `Like`
- `entity/Amenity/` — shared spaces, schedules, exceptions, bookings
- `entity/Timeline/` — service/project tracking with steps, staff assignments, attachments
- `entity/Outer/` — gatehouse records: `Visitor`, `Delivery`

Repositories and controllers mirror the same package structure.

## Domain Overview

This is a Brazilian condominium management system. Key concepts:

- **Residents** can create posts, book amenities, pre-authorize visitors, and receive notifications.
- **Posts** use single-table inheritance. The discriminator column determines the concrete type (ticket, suggestion, poll, trade, notice).
- **Tickets** track maintenance/issue requests with status flow and category (both defined as enums in Portuguese).
- **ServiceTimeline** tracks multi-step services; steps can have assigned staff and photo/document attachments.
- **Bookings** are validated against `AmenitySchedule` (weekly hours) and `AmenityException` (holidays/closures).
- **Notifications** are generated by DB triggers for deliveries and visitor arrivals; other events are application-driven.

## Project State

The project is a scaffold: entities, repositories, and controllers are defined but most service and controller method bodies are empty. Security/authentication has not been implemented yet.
